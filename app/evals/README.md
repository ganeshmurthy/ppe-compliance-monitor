# Evals

Evaluates backend endpoints by sending predefined inputs and scoring responses.

Supports two features via the `EVAL_FEATURE` env var:

| Feature | Default | Endpoint | Judging |
|---------|---------|----------|---------|
| `chat`  | yes     | `/api/chat` | GEval LLM judge (Correctness) |
| `alerts`| no      | `/alerts`   | SQL result comparison (deterministic) |

`EVAL_DATASET` selects the sub-dataset (default: `ppe`). Multiple app configs are supported (PPE, bird, yolo).

## File layout

```
app/evals/
  db_seed_data.sql              # Full DB snapshot (all app_configs), generated by init-eval-db
  datasets/
    chat/
      ppe/
        eval_dataset.json       # PPE chat questions + golden answers
      bird/
        eval_dataset.json
      yolo/
        eval_dataset.json
    alerts/
      ppe/
        eval_dataset.json       # PPE alert rules + golden SQL/results
  preds/
    chat/                       # Chat eval output
    alerts/                     # Alert eval output
```

- `db_seed_data.sql` is shared across all features and datasets.
- Each `datasets/<feature>/<dataset>/` contains manually curated JSON files.
- `EVAL_FEATURE` selects the feature (default: `chat`).
- `EVAL_DATASET` selects which dataset to run within that feature (default: `ppe`).

## Prerequisites

The full local stack must be running before running evals:

```bash
make local-build-up
```

OpenAI env vars (`OPENAI_API_TOKEN`, `OPENAI_API_ENDPOINT`, `OPENAI_MODEL`) must be set in `.env`.

## Snapshot the database

Before running evals for the first time (or after the live data changes significantly), snapshot the running database:

```bash
make init-eval-db
```

This dumps `app_config`, `detection_classes`, `detection_tracks`, and `detection_observations` into `app/evals/db_seed_data.sql`.

## Run evals

```bash
# Default (chat, PPE)
make eval

# Chat with specific dataset
EVAL_DATASET=bird make eval

# Alert evals
EVAL_FEATURE=alerts make eval

# Alert evals with specific dataset
EVAL_FEATURE=alerts EVAL_DATASET=ppe make eval
```

The eval script automatically:

1. Saves the live database to a volume-backed snapshot file
2. Loads the eval seed data from `db_seed_data.sql`
3. Runs the experiments from `datasets/<feature>/<dataset>/*.json`
4. Restores the original live data from the snapshot (in a `finally` block)

## Dataset schemas

### Chat (`datasets/chat/<dataset>/eval_dataset.json`)

```json
[
  {
    "id": "unique_test_id",
    "question": "How many workers are wearing hardhats?",
    "description": "Hardhat: 3, Person: 5",
    "golden_answer": "3 workers are wearing hardhats out of 5 people."
  }
]
```

### Alerts (`datasets/alerts/<dataset>/eval_dataset.json`)

```json
[
  {
    "id": "unique_test_id",
    "rule": "Alert me if more than 3 people are without hardhats in the last hour",
    "golden_sql": "SELECT COUNT(*) FROM detection_observations ...",
    "golden_result": "5"
  }
]
```

`app_config_id` is resolved automatically from the dataset name (ppe=2, bird=1, yolo=3).

## Adding a new dataset

1. Create a directory under `datasets/<feature>/` with your dataset name.
2. Add an `eval_dataset.json` with entries matching the schema above.
3. Make sure `db_seed_data.sql` contains relevant data (run `make init-eval-db`).
4. Run: `EVAL_FEATURE=<feature> EVAL_DATASET=<name> make eval`
